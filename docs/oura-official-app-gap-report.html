Official Oura App vs Cracked Oura Desktop and Mobile

Gap analysis of the current official Oura customer experience against this repository's Electron desktop app, Android mobile app, and local backend. Researched and prepared on May 22, 2026.

Official sources checked: Oura Help, App Store Local surfaces checked: frontend, android-mobile, backend Repo status: alpha, export-driven data model
Strong

Historical exported-data analysis, local privacy, custom dashboards, trends, and AI analyst workflows.

Partial

Sleep, readiness, activity, resilience, cardiovascular age, workouts, tags, battery, and mobile summaries.

Weak

Live ring/device control, guided logging, official algorithms, and customer lifecycle features.

Executive Summary

Cracked Oura is best positioned as a privacy-first, no-subscription analytics layer over exported Oura data. It already competes well on longitudinal analysis, customizable dashboards, local storage, and free-form AI interrogation of personal history. It does not yet match the official app as a full companion app because the official app owns ring setup, Bluetooth sync, live device settings, action cards, guided logging, reports, experiments, and newest personalized health features.

What Cracked Oura Provides Today

The project currently works as a local data warehouse plus desktop and Android clients. It is export-first, not Bluetooth-first: the backend automates or manually ingests Oura data exports, then the apps visualize and analyze the local database.

Local product area Implemented capability Current limits
Desktop dashboard React/Electron app with Today, Sleep, Readiness, Activity, Stress & Resilience, Trends & Explorer, Tags & Journal, AI Analyst, and custom dashboards. Many views depend on exported data. It does not control the ring or provide live in-day sync.
Data ingestion FastAPI backend imports Oura export ZIP files and stores sleep, activity, readiness, resilience, sleep sessions, workouts, meditation, battery, heart rate, temperature, ring configuration, tags, and cardiovascular age. Export cadence can lag by hours or days. Some official-only derived features are not present in export data.
Custom analytics Dynamic query API, schema introspection, reusable widgets, table/chart/gauge/radar widgets, and dashboard layout editing. Advanced correlation, anomaly detection, and cohort-style explanations are still limited.
AI Analyst Persistent chat threads and an LLM-backed analyst that can query local history. Needs stricter health disclaimers, structured tool outputs, traceable cited metrics, and safer recommendation boundaries.
Android app Native Kotlin/Compose app with Today, Sleep, Trends, AI, Settings, manual sync, offline Room cache, day detail, local sleep-debt approximation, baseline deltas, and workouts. Manual snapshot sync only; summary/window data rather than full high-frequency streams; no background sync yet.
Privacy and ownership Local SQLite database, local mobile sync API, token-protected Android sync, and no subscription dependency for viewing imported/exported data. Security posture needs hardening before broad distribution: token lifecycle, encrypted storage, HTTPS/Tailscale guidance, backups, and threat-model docs.

Highest-Impact Gaps To Close

1. Make the product honest about its role

Position it as "local analytics for your exported Oura history," not a full replacement companion app.

  • Add copy in onboarding explaining export freshness, unsupported live ring controls, and privacy benefits.
  • Show "last Oura export" and "latest local day" prominently in desktop and Android.

2. Add guided health detail views

The official app wins because it explains scores. Cracked Oura stores enough data to make this much better.

  • Normalize sleep/readiness/activity contributors into reusable UI cards.
  • Add "why this changed" deltas against 7/14/30-day baselines.

3. Build local action cards

Close the Today-tab gap without needing official action feeds.

  • Cards for low battery, stale sync, low readiness, sleep debt, HRV drop, temperature anomaly, inactivity, and late bedtime.
  • Make each action traceable to the data point that triggered it.

4. Ship mobile background sync

Manual mobile sync makes the Android app feel like a viewer. Background refresh makes it feel like a product.

  • Use WorkManager for scheduled token-protected sync.
  • Add sync conflict/failure notifications and a compact status widget.

5. Add reports

Reports are a high-leverage desktop feature and align with a local-data product.

  • Generate weekly/monthly HTML and PDF reports from the local database.
  • Include trends, anomalies, workouts, tags, sleep regularity, and AI-written summaries with cited dates.

Recommended Roadmap

Scope is intentionally limited to the two workstreams below: parity polish for the daily experience, then differentiation through local historical analysis, explainable AI, reports, and dashboard templates.

Phase Goal Deliverables Why now
Phase 1: Parity polish Make stored Oura export data as understandable as the official app. Contributor cards, better sleep/readiness/activity detail, local action cards, sync freshness UI, mobile background sync. Uses data already present in the database and closes daily-use friction.
Phase 2: Differentiation Lean into what official Oura does not do well: local history and custom analysis. Correlation explorer, anomaly detection, AI citations, saved investigations, weekly/monthly reports, dashboard templates. This is the project's natural advantage and does not require ring control.

Phase 1 Detailed Specification: Parity Polish

Phase 1 should make the apps feel dependable for daily use with the data already present in the local database. It should not introduce external integrations or unofficial ring control.

Phase 1 Objectives

Objective Spec Acceptance criteria
Explain core scores Normalize sleep, readiness, and activity contributors into a shared contributor model with label, value, direction, status, short explanation, and source field path. Render the same contributor cards on desktop detail views and Android day detail. Every day with contributor JSON shows contributor cards for all available domains. Missing contributors show a compact "not exported for this day" state, not blank UI.
Improve daily guidance Replace generic daily text with deterministic guidance generated from readiness, sleep, activity, HRV, resting HR, sleep debt approximation, temperature deviation, and activity goal progress. Guidance cites the exact metric that triggered it, for example "HRV is 8 ms below your 14-day baseline." No advice is shown without a data reason.
Add local action cards Create a rules engine that emits prioritized cards for stale sync, low battery, low readiness, poor sleep, high temperature deviation, HRV drop, high stress/recovery imbalance, missed activity target, and missing data. Today view renders 0-5 cards sorted by severity. Each card has title, reason, suggested action, source metrics, and dismiss state for the current day.
Show sync freshness Surface latest local Oura day, last export request, last successful ingest, current automation status, and mobile sync server state in one shared status component. Desktop sidebar/top bar and Android Settings/Today show whether data is fresh, stale, syncing, or blocked by OTP/login/server configuration.
Background Android sync Add WorkManager scheduled sync using the existing token-protected desktop mobile API. Respect the configured sync window and preserve manual "sync now" behavior. User can choose off/manual, every 6 hours, every 12 hours, or daily. Last background result is visible with failure diagnostics already supported by the repository.

Phase 1 Backend/Data Contracts

Contract Fields Notes

Phase 1 UI Requirements

Surface Required changes
Desktop Today Add action cards below the greeting and above core scores. Add a compact sync freshness chip. Replace generic greeting copy with deterministic daily guidance from the backend read model.
Desktop Sleep/Readiness/Activity Add contributor sections, baseline deltas, trend mini-charts, and evidence links back to the raw field path or dashboard metric. Keep existing visual language and avoid adding a new navigation model.

Phase 1 Non-Goals

  • No third-party platform integrations, watch companion, or widget work.
  • No medical symptom prediction features.
  • No Bluetooth ring setup, firmware updates, airplane mode, live HR recording, or workout GPS tracking.
  • No attempt to exactly reproduce proprietary Oura algorithms when export data does not expose them.

Phase 2 Detailed Specification: Differentiation

Phase 2 should lean into the project's advantage: local historical data, custom analysis, and explainable AI. The product should become better than the official app for people who want to inspect patterns across months or years.

Phase 2 Objectives

Objective Spec Acceptance criteria
Correlation Explorer Add an analysis view that compares two metrics over a selected date range with lag controls, scatter plot, trend overlays, Pearson/Spearman correlation, sample count, missing-data warnings, and plain-English interpretation. User can answer questions such as "does later bedtime affect readiness next day?" with a saved analysis and reproducible metric/date selections.
Anomaly detection Compute local anomalies for HRV, resting HR, temperature deviation, sleep duration, sleep timing, activity, stress/recovery share, and readiness. Use rolling baselines and robust z-score or percentile bands. Trends and Today can highlight unusual days with the exact baseline window, threshold, and metric evidence. Anomalies are labeled "local anomaly," not diagnosis.
AI with citations Upgrade analyst responses to include structured evidence cards: metric, date range, aggregate, comparison, source query, and confidence. Text answers should cite those cards. Every health-data claim in an AI answer can be traced to a query result or explicit "insufficient data" statement. Unsupported medical claims are refused or softened.
Saved investigations Let users save an analysis session containing selected metrics, filters, date range, chart config, AI prompt, generated answer, and evidence cards. Saved investigations appear in the AI Analyst and Trends sections, can be renamed/deleted, and remain stable across app restarts.
Reports Generate weekly and monthly HTML reports from local data. Include score trends, anomalies, best/worst days, sleep consistency, activity target adherence, workouts, tags, and optional AI summary with citations. Desktop can export a self-contained HTML report. PDF export is optional if it requires additional packaging work.
Dashboard templates Add predefined dashboards for Sleep Quality, Recovery, Training Load, Stress & Resilience, and Long-Term Health. Templates should instantiate existing widget types and metric paths. A user can create a useful dashboard in one click without configuring widgets manually. Templates remain editable.

Phase 2 Backend/Data Contracts

Contract Fields Notes

CorrelationResult x_metric, y_metric, lag_days, method, coefficient, sample_count, p_value_optional, interpretation Lag is essential for Oura use cases, especially sleep tonight vs readiness tomorrow.
AnomalyResult metric, day, value, baseline, score, direction, severity, method Persisting is optional. Computing on demand is acceptable if performance stays acceptable over multi-year data.

ReportDefinition id, period, start_day, end_day, sections, generated_at, html Reports can be regenerated; stored HTML is useful for stable sharing and snapshots.

Phase 2 UI Requirements

Surface Required changes
Desktop Trends & Explorer Add compare mode with metric A/B selectors, lag selector, scatter plot, line overlay, correlation panel, missing-data warning, and "save investigation" action.
Desktop AI Analyst Add evidence card rendering below responses, suggested follow-up charts, "save as investigation," and refusal language for unsupported medical advice. Keep chat thread persistence.
Desktop Reports Add a Reports view or Tools subsection for weekly/monthly report generation. Reports should preview in-app and export to a self-contained HTML file.
Desktop Dashboards Add template picker during dashboard creation and in the dashboard editor. Templates should use existing widget registry entries where possible.
Android Keep Android Phase 2 narrower: display saved investigations summaries, anomaly markers in Trends/Day Detail, and read-only report summaries. Do not attempt full report authoring on mobile.

Phase 2 Analysis Rules

  • Every statistical result must include sample count and date range.
  • Correlation views must warn when fewer than 14 paired samples are available and should hide interpretation below 7 samples.
  • Lagged comparisons must label direction clearly: metric A on day D compared with metric B on day D+N.
  • Anomaly detection must use rolling personal baselines, not global norms.
  • AI claims must be grounded in evidence cards. If evidence is missing, the answer should say what data is missing.

Phase 2 Non-Goals

  • No third-party ingestion or export beyond local report HTML.
  • No clinical diagnosis, disease prediction, or medication guidance.
  • No automatic push notifications for anomalies unless explicitly moved back into scope later.
  • No major rewrite of the dashboard system; templates should compose existing widgets first.

Implementation Notes By Code Area

Area Suggested changes
backend Add derived-metrics services for contributors, anomalies, sleep debt, cumulative stress approximation, and report generation. Keep raw imports unchanged; store derived results in versioned tables or compute on read with explicit algorithm labels.
frontend Add reusable contributor components, local action-card engine, report export view, Labs view, and clearer device/sync status. Expand AI Analyst to return cited metric cards alongside text.
android-mobile Add WorkManager background sync, richer detail screens, anomaly markers, saved-investigation summaries, and selective high-frequency heart-rate and temperature sync for charts and anomaly detection.
security/privacy Encrypt local mobile token storage, document LAN/Tailscale threat model, add token rotation, support HTTPS reverse-proxy guidance, and make destructive data actions explicit.

Risks And Constraints

  • Official algorithms are not fully reproducible. Sleep need, Symptom Radar, and cumulative stress should be labeled as local approximations unless sourced directly from exported data.
  • Ring control is not a good near-term target. Firmware, airplane mode, setup, find-ring, and live Bluetooth behavior are likely proprietary and fragile.
  • Mobile freshness matters. Without background sync and notifications, the Android app will feel behind the official app even if the charts are better.
  • Health advice needs guardrails. AI output should cite data, avoid diagnosis, and recommend professional care for concerning symptoms.
